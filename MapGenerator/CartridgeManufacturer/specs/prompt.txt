You are implementing the next core phase of the MapGenerator project.

Your task is to design and implement the following three components:

1. A Rust crate named `wcar` that defines the WorldCartridge (.wcar) binary format
2. A WCAR → StarCraft CHK exporter
3. A headless Stratagus runner for simulation and validation

This work is based on the WCAR + CHK v0.2 specification.


────────────────────────────────────────
HIGH-LEVEL GOALS
────────────────────────────────────────

• WCAR is the canonical world artifact
• CHK is a derived executable projection
• Stratagus is used as a real RTS engine for validation
• The system must support deterministic, automated testing

WCAR → CHK → Stratagus → simulation

────────────────────────────────────────
DELIVERABLES
────────────────────────────────────────

1. Rust crate: `wcar`
   - Read/write WCAR files
   - Chunk-based binary format
   - Safe skipping of unknown chunks
   - Strong typing for standard chunks
   - Zero external runtime dependencies beyond serde (if needed)

2. Rust binary: `wcar_export_chk`
   - Input: `.wcar`
   - Output: `.chk` or `.scx`
   - Maps WCAR chunks to StarCraft CHK sections
   - Projection is lossy by design
   - CHK output must be valid and loadable by Stratagus/Stargus

3. Rust binary: `wcar_run_stratagus`
   - Runs Stratagus headlessly
   - Loads exported CHK
   - Executes a short simulation
   - Reports success/failure via exit code
   - Designed for future AI-driven regression testing

────────────────────────────────────────
IMPORTANT CONSTRAINTS
────────────────────────────────────────

• WCAR is immutable once written
• WCAR is never generated from CHK
• CHK is treated as opaque binary data inside WCAR
• Use little-endian encoding
• Use fixed-width numeric types
• Prefer explicit structs over generic maps
• Avoid premature abstraction

────────────────────────────────────────
IMPLEMENTATION GUIDANCE
────────────────────────────────────────

• Follow the WCAR + CHK v0.2 specification exactly
• Separate parsing, validation, and projection layers
• Favor correctness and clarity over performance
• Code must be readable by future maintainers
• Add comments where binary layout matters

• No UI work
• No HTML output
